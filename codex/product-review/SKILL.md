# 产品评审（公开版）

按“先还原闭环，再找落地漏洞”的方式评审需求。不要停留在措辞润色，要优先暴露业务目标不清、范围失控、权限漏口、状态不闭环、交互缺规则、数据口径不一致、异常场景缺失和上线依赖不明的问题。

## 适用场景

- 审查 `PRD`、需求方案、功能设计、原型说明、评审材料、上线方案。
- 评审中后台、平台型、机构侧、直播后台、作者体系、权限状态流转类需求。
- 在进入开发前识别会阻塞交付、联调、测试、上线的关键风险。

## 不适用场景

- 单纯润色文案或改写表达。
- 以创意发散为主的脑暴讨论。
- 市场分析、商业战略、定价策略等非需求评审任务。

## 接收输入

- 优先读取 `PRD`、功能说明、原型标注、页面清单、流程图、状态图、字段表、埋点方案、上线计划。
- 先判断材料是否足够支撑评审；如果关键信息缺失，明确写出“待补充信息”，不要强行假设关键规则。
- 如果输入同时包含多个模块，先按模块拆分，再在结论中给总风险判断。

## 先还原需求骨架

在正式评审前，先用简短语言提炼以下信息，作为后续判断基础：

- 需求要解决什么业务问题，服务什么目标。
- 涉及哪些角色、系统、页面、流程节点。
- 哪些内容在本期范围内，哪些明确不做。
- 哪些规则已经明确，哪些仍是待确认项。

如果以上四项无法提炼完整，直接在评审结论中标出信息不足。

## 逐项评审维度

### 1. 需求背景与业务目标

- 检查为什么现在做这件事，是否给出明确业务背景。
- 检查目标是否可验证，例如提升转化、降低人工成本、减少违规、提升履约效率。
- 检查是否定义成功标准、观察口径、验收标准。
- 检查是否存在“只写功能，不写业务价值”的情况。

### 2. 需求范围与边界

- 检查本期范围、非本期范围、灰度范围、适用对象是否写清。
- 检查是否区分新流程、兼容旧流程、迁移历史数据。
- 检查上下游影响范围是否明确，包括机构端、作者端、运营端、审核端、直播后台、配置后台。
- 检查是否存在默认兜底逻辑缺失，导致实现人员自行补规则。

### 3. 角色与权限

- 检查角色定义是否完整，包括谁可见、谁可操作、谁可审批、谁可导出、谁可配置。
- 检查是否区分组织权限、数据权限、按钮权限、状态驱动权限。
- 检查跨角色协作时的责任边界，例如提交方、审核方、回退方、配置方。
- 检查越权、误操作、代操作、权限继承和角色切换后的表现。

### 4. 主流程、分支流程、状态流转

- 检查主流程是否闭环，从入口、处理中、完成、失败、回退到终态是否连贯。
- 检查所有关键分支，例如驳回、撤回、超时、重复提交、并发处理、配置变更后重试。
- 检查状态枚举、状态进入条件、状态退出条件、状态展示文案是否一致。
- 检查状态流转是否允许逆向操作，逆向后数据和页面如何表现。

### 5. 页面元素与交互逻辑

- 检查页面入口、列表、详情、弹窗、批量操作、筛选、导出、提示语是否完整。
- 检查字段是否具备名称、类型、是否必填、默认值、校验规则、可编辑条件。
- 检查交互是否说明按钮置灰、不可操作提示、保存与提交区别、离开页面后的数据处理。
- 检查列表和详情是否口径一致，前后页面是否可能出现信息不一致。

### 6. 数据定义、字段口径、埋点报表

- 检查核心对象、字段定义、状态字段、统计口径是否明确。
- 检查是否说明数据来源、刷新机制、延迟容忍度、历史数据处理方式。
- 检查是否给出埋点、日志、报表、看板或审计字段要求。
- 检查指标是否可归因，能否支持上线后验证成败。

### 7. 异常场景与边界条件

- 检查空数据、脏数据、历史遗留数据、外部接口失败、重复点击、并发冲突、网络异常。
- 检查边界值，如数量上限、字符长度、时间边界、权限过期、状态穿透。
- 检查人工兜底、补偿机制、告警方式和错误提示策略。
- 检查是否明确失败后谁处理、在哪处理、能否重试。

### 8. 上线依赖与项目风险

- 检查上下游系统、接口、配置、数据初始化、运营流程、培训公告是否是上线前置依赖。
- 检查是否定义灰度策略、开关策略、回滚策略、兼容旧版本方案。
- 检查是否存在需求频繁变动、规则待定、接口未定、资源不足等项目风险。
- 检查测试重点、联调重点、验收门槛是否明确。

## 边界场景专项评审

当需求涉及摘要、概览、AI 生成内容、聚合结论、推荐理由、状态提示或自动化处理结果时，必须按以下五类边界补充评审。不要只写"异常时提示失败"，要明确哪些场景不展示摘要，哪些场景降级展示，哪些场景允许用户或系统重试。

### 1. 内容侧边界

| 可能出现的边界场景 | 建议的处理方式 | 是否需要产品在 PRD 中明确写出 |
| --- | --- | --- |
| 内容为空、正文缺失、只有标题、只有封面图或附件不可解析 | 不展示摘要；展示空态或"暂无可摘要内容"；不触发摘要生成消耗 | 必须明确 |
| 内容过短，低于摘要生成阈值，例如少于指定字数、少于有效段落数 | 不展示摘要或直接展示原文片段；阈值、判断字段和前端文案需写清 | 必须明确 |
| 内容正在编辑、草稿未发布、审核中、被驳回、已下架、已删除 | 默认不展示摘要；如果运营后台需要排查，可仅在有权限的详情页展示状态原因 | 必须明确 |
| 内容包含敏感信息、违规词、未成年人隐私、联系方式、价格或合同等高风险信息 | 不生成或不展示摘要；必要时展示"因内容安全规则暂不可展示"；记录风控原因供后台排查 | 必须明确 |
| 内容来源为外链、第三方接口、转码文件、直播回放、会议纪要等异步内容 | 解析完成前降级展示标题、封面、状态；解析失败可允许系统重试，超过次数后停止并暴露失败原因 | 必须明确 |
| 内容部分缺失，例如图片加载失败、音视频转写不完整、章节缺失 | 可降级生成"基于已解析内容的摘要"，但必须标记摘要不完整；核心内容缺失时不展示摘要 | 必须明确 |
| 内容更新后摘要仍是旧版本 | 摘要需失效并重新生成；重新生成期间展示旧摘要还是生成中状态需要二选一写清 | 必须明确 |
| 摘要置信度低、模型返回空结果、结果明显重复或无意义 | 不展示摘要；允许后台或系统自动重试；前台不要展示低质量占位文案 | 必须明确 |

### 2. 用户侧边界

| 可能出现的边界场景 | 建议的处理方式 | 是否需要产品在 PRD 中明确写出 |
| --- | --- | --- |
| 未登录、登录态过期、账号被封禁或注销 | 不展示摘要内容；引导登录或展示无权限状态；登录态恢复后可重新请求 | 必须明确 |
| 用户无内容查看权限，例如非本机构、非班级成员、非项目参与人、非审核员 | 不展示摘要，不能通过摘要泄露原文信息；返回无权限或隐藏入口 | 必须明确 |
| 用户只有部分数据权限，例如只能看本人、所管机构、所带班级 | 摘要只能基于用户可见数据生成；不能使用全量数据生成后再截断展示 | 必须明确 |
| 新用户、无历史行为、无绑定对象、无订阅内容 | 降级为空态、默认推荐或基础说明；不展示个性化摘要 | 必须明确 |
| 用户关闭个性化、拒绝授权、隐私设置不允许加工内容 | 不展示个性化摘要；可展示非个性化公共摘要；需说明开关影响范围 | 必须明确 |
| 多端登录或角色切换后权限变化 | 重新校验摘要可见性；角色变化后旧摘要缓存必须失效或重新拉取 | 必须明确 |
| 用户主动刷新、重新生成、手动重试摘要 | 仅在生成失败、超时、内容已更新或用户有操作权限时允许重试；需限制频率和次数 | 必须明确 |
| 用户举报、拉黑、屏蔽某内容或某作者 | 不展示相关内容摘要；列表和详情口径一致 | 必须明确 |

### 3. 状态侧边界

| 可能出现的边界场景 | 建议的处理方式 | 是否需要产品在 PRD 中明确写出 |
| --- | --- | --- |
| 摘要状态为未生成、生成中、生成成功、生成失败、已过期、已禁用 | 状态枚举、进入条件、退出条件、页面文案和按钮状态必须完整定义 | 必须明确 |
| 内容状态与摘要状态不一致，例如内容已下架但摘要仍成功 | 以内容可见状态优先；内容不可见时不展示摘要，并清理或屏蔽缓存 | 必须明确 |
| 生成中时间过长或异步任务排队 | 降级展示"生成中"或原内容信息；超过超时时间后进入失败态，可自动或手动重试 | 必须明确 |
| 审核通过后才允许展示摘要 | 审核前不展示摘要；审核通过后触发生成或开放展示；驳回后摘要不可见 | 必须明确 |
| 内容被撤回、回滚版本、重新发布 | 摘要版本需与内容版本绑定；回滚后展示对应版本摘要或重新生成 | 必须明确 |
| 批量生成、批量刷新、批量失败 | 允许部分成功；列表需区分成功、失败、处理中，不应用一个失败阻断全部结果 | 必须明确 |
| 灰度、开关关闭、实验分组未命中 | 不展示摘要入口或展示原有页面；不要展示半功能状态 | 必须明确 |
| 运营后台手动干预，例如禁用摘要、置顶摘要、人工改写 | 前台以人工状态优先；需记录操作人、时间、原因和恢复方式 | 必须明确 |

### 4. 异常侧边界

| 可能出现的边界场景 | 建议的处理方式 | 是否需要产品在 PRD 中明确写出 |
| --- | --- | --- |
| 网络超时、接口 5xx、网关限流、模型服务不可用 | 前台降级展示原页面核心信息；摘要区域展示短失败态或隐藏；允许稍后重试 | 必须明确 |
| 接口 4xx，例如参数错误、内容不存在、权限不足 | 权限不足和内容不存在不允许重试；参数错误进入排查日志，不让用户反复点击 | 必须明确 |
| 第三方解析、转写、OCR、模型生成失败 | 可系统自动重试；重试次数、间隔、最终失败状态和后台处理入口需写清 | 必须明确 |
| 重复点击生成、刷新、提交 | 按钮置灰或展示处理中；后端需幂等，避免重复扣费、重复任务、重复通知 | 必须明确 |
| 并发更新，例如内容编辑和摘要生成同时发生 | 以最新内容版本为准；旧任务完成后不得覆盖新摘要 | 必须明确 |
| 缓存命中异常或数据延迟 | 可短期展示旧摘要但必须有版本或更新时间约束；涉及强一致场景时不展示旧摘要 | 必须明确 |
| 摘要生成成本过高或达到配额上限 | 降级为不生成或排队；可提示"暂不可生成"；后台需有配额告警 | 必须明确 |
| 前端渲染异常，例如摘要过长、特殊字符、换行、富文本污染 | 截断、转义、限制最大行数；禁止富文本污染页面；原文仍可正常访问 | 必须明确 |

### 5. 权限/风控侧边界

| 可能出现的边界场景 | 建议的处理方式 | 是否需要产品在 PRD 中明确写出 |
| --- | --- | --- |
| 摘要可能泄露用户无权查看的正文、评论、附件、学员信息、交易信息 | 摘要生成前按用户或角色权限过滤数据；无权内容不得进入摘要输入 | 必须明确 |
| 管理员、运营、机构、老师、学生、家长等角色看到的摘要范围不同 | 按角色定义摘要来源、字段脱敏规则和入口可见性；列表、详情、导出需一致 | 必须明确 |
| 内容命中安全审核、版权风险、广告导流、涉政涉黄涉暴等规则 | 不展示摘要；可展示安全拦截状态；后台保留原因和申诉/复核入口 | 必须明确 |
| 摘要涉及个人信息、未成年人数据、联系方式、身份证、手机号、地址 | 默认脱敏或不展示；PRD 需写清脱敏字段、脱敏格式和例外权限 | 必须明确 |
| 用户频繁触发生成、批量刷接口、疑似爬取摘要 | 限流、冷却、验证码或后台告警；达到风控阈值后不允许继续重试 | 必须明确 |
| 账号角色刚被移除、组织关系刚解除、课程或机构到期 | 立即按最新权限隐藏摘要；缓存需失效，不能继续展示历史摘要 | 必须明确 |
| 审计要求追踪摘要生成、查看、导出、人工修改 | 记录操作人、对象、时间、来源、生成版本、失败原因和导出行为 | 必须明确 |
| 人工编辑摘要与 AI 摘要并存 | 明确优先级、标识、回滚、审核和责任归属；高风险场景需先审后发 | 必须明确 |

评审输出时，如果 PRD 没有覆盖以上任一适用项，必须把缺口写入"高优先级问题"或"文档缺失项清单"。其中涉及权限泄露、状态穿透、风控拦截、内容不可见但摘要可见、旧摘要覆盖新内容的情况，默认按高优先级问题处理。

## 问题分级标准

### 高优先级问题

归入高优先级的情况：

- 会直接阻塞开发、测试、联调或上线。
- 会造成业务闭环断裂、权限漏洞、状态错误、数据错账、统计失真。
- 会导致多个角色理解不一致，开发只能自行补规则。
- 会在上线后产生明显客诉、运营事故、履约异常或审计风险。

每条高优先级问题都必须写清：

- 问题描述
- 影响范围
- 为什么是问题
- 建议如何补充/修改

### 中低优先级问题

归入中低优先级的情况：

- 不一定阻塞开发，但会放大返工成本或造成体验割裂。
- 规则基本可实现，但表达不完整、页面交互不统一、数据说明不充分。
- 对后续运营、分析、培训、扩展造成隐性成本。

每条中低优先级问题都必须写清：

- 问题描述
- 风险说明
- 优化建议

## 风险判断口径

- 低风险：闭环完整，关键规则基本明确，仅有少量表达或补充项。
- 中风险：可进入详细评审，但存在多个需要补齐的边界、权限或数据口径问题。
- 高风险：关键目标、范围、流程、权限、状态或上线依赖不明确，不建议直接进入开发。

## 输出要求

- 使用正式、专业、可复制到评审意见区的中文表达。
- 明确指出“待补充信息”，不要用模糊判断掩盖信息缺失。
- 优先指出逻辑漏洞和落地风险，不要只给润色建议。
- 如果问题很多，优先输出最影响闭环和上线的部分。
- 如果文档质量较高，也要指出仍需确认的风险点，不要只给笼统正面评价。
